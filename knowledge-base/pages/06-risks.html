Module 06 Risk Analysis

Risks, Gaps & Recommendations

What can go wrong, what's missing, and what to do about it — a risk-driven roadmap.

🚨 Risk Register

▶ HIGH R1: Loop Interference
Click to expand

Description

Outer loop (prompt optimization) and inner loop (weight optimization) interfere when running concurrently. Prompts optimized for model v1 may degrade performance on model v2.

Impact

Wasted compute cycles, oscillating quality, inability to attribute improvements to either loop. Could undermine the entire two-loop design.

Mitigation

Version-lock model weights during outer-loop passes. Run loops on alternating schedules (e.g., outer loop daily, inner loop weekly). Gate weight updates on eval regression checks. Maintain a "known-good" config snapshot.

Residual Risk

Medium — scheduling discipline reduces but doesn't eliminate the risk. Prompt–weight co-adaptation remains an open research question.

▶ HIGH R2: Eval Noise
Click to expand

Description

LLM judge scores have inherent variance from temperature, prompt sensitivity, and non-deterministic generation. Single-run eval results can be misleading.

Impact

False positives (shipping a regression) or false negatives (rejecting a good model). Training decisions based on noise rather than signal.

Mitigation

3x runs per eval, take median score. Report variance. If variance > 15%, investigate judge prompt or increase sample size. Use CODE judges (deterministic) alongside LLM judges.

Residual Risk

Low — 3x median is well-validated. Remaining risk is systematic judge bias, addressable via calibration against human review.

▶ HIGH R3: Tool-Use Fidelity
Click to expand

Description

Fine-tuned open models may generate syntactically correct but semantically incorrect tool calls — wrong arguments, hallucinated tool names, or invalid schemas.

Impact

Silent failures in production — agent appears to work but produces wrong results. Especially dangerous for stateful tools (file writes, API calls, cluster mutations).

Mitigation

Include tool-call-specific eval judges (schema validation, arg completeness, call sequence checks). Use constrained decoding for tool calls. Include tool-call examples prominently in SFT data.

Residual Risk

Medium — constrained decoding helps syntax but not semantics. Semantic tool-use correctness remains hard to evaluate automatically.

▶ MEDIUM R4: Cross-Provider Legality
Click to expand

Description

Using Claude Opus outputs as training data for open-weight models may violate Anthropic's Terms of Service regarding model distillation and competitive training.

Impact

Legal exposure. May need to switch frontier model to one with permissive output licensing (e.g., use Nemotron or self-hosted frontier model).

Mitigation

Legal review of Anthropic ToS and output ownership clauses. Consider using only self-hosted models for trace generation. Evaluate Nemotron or other permissive alternatives as the frontier model.

Residual Risk

Low if using self-hosted frontier model. Medium if relying on Claude outputs for training data.

▶ MEDIUM R5: Multi-Turn Credit Assignment
Click to expand

Description

Agent tasks span many turns with tool calls. A final success/failure signal at the end of a trajectory doesn't tell the model which specific turns were good or bad.

Impact

Noisy reward signals. SFT treats all turns equally; DPO/GRPO need turn-level or sub-trajectory preferences. Training may reinforce bad intermediate steps.

Mitigation

Use per-span judge annotations (MLflow assessments at span level, not just trace level). Implement step-level reward modeling. Consider MCTS-style trajectory exploration for better credit assignment.

Residual Risk

Medium — this is an active research area. Practical per-step judging is expensive and noisy.

▶ MEDIUM R6: EvalHub Adapter
Click to expand

Description

The agent-eval-harness needs to plug into EvalHub as a provider. No adapter exists — the harness uses its own CLI and config format (eval.yaml) that differs from EvalHub's orchestration model.

Impact

Without the adapter, agent skill evaluation runs separately from model evaluation (lm-evaluation-harness) and safety evaluation (Garak). No unified dashboard. Manual coordination required.

Mitigation

Build a thin EvalHub provider that wraps eval-harness CLI calls and publishes results to the shared MLflow experiment. Start with a script-based integration; formalize later.

Residual Risk

Low — engineering effort, not a fundamental challenge. 1–2 sprint effort.

▶ LOW-MED R7: Trace Schema Gap
Click to expand

Description

MLflow traces use OpenTelemetry hierarchical span format. Training pipelines need flat JSONL with messages arrays. A converter must be built to bridge this gap.

Impact

Blocks all training data preparation. Without the converter, traces cannot be used for SFT/DPO/GRPO regardless of volume.

Mitigation

Build trace-to-training converter as a standalone Python package. Start with SFT format (simplest), extend to DPO/GRPO. Include unit tests with real trace samples.

Residual Risk

Low — well-scoped engineering task. Main risk is edge cases in span tree traversal (recursive tools, parallel branches).

4. Eval Reliability

How to ensure eval results are trustworthy?

Protocol3x runs, median score, variance report
Threshold< 15% variance to trust the result
BackupCODE judges as deterministic anchors

5. Loop Scheduling

How to coordinate the two optimization loops to avoid interference?

Outer Loop

Run continuously (hourly/daily). Version-lock model weights. No concurrent inner loop runs allowed.

Inner Loop

Run periodically (daily/weekly). Pause outer loop during weight update. Eval gate before traffic shift.

Coordination

Shared state via MLflow tags. Outer loop reads model version tag. Inner loop sets "training-in-progress" flag.
