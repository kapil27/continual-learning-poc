The Two-Loop Architecture

A unified system that improves AI agents autonomously from their own production experience — two nested optimization loops connected by MLflow as the shared data plane.

2 Loop Comparison

🎯

What It Optimizes

What the agent is told — skill instructions, prompts, and tool configurations stored in SKILL.md files.

⚡

Speed & Cost

Runs in minutes per iteration. Low cost — only API calls + judge evaluations. No GPU training required.

📦

Output

Improved SKILL.md files — better instructions, refined prompts, optimized tool configurations.

🔧

Key Tool

agent-eval-harness — eval-analyze → eval-dataset → eval-run → eval-review → eval-optimize

🎯

What It Optimizes

How the model performs — updates model weights based on production traces via SFT, DPO, and GRPO training.

⚡

Speed & Cost

Runs in hours to days per iteration. High cost — requires GPU compute for training. Runs periodically (daily/weekly).

📦

Output

Improved model checkpoints — better weights deployed via KServe canary with traffic splitting.

🔧

Key Tools

Training Hub (Kubeflow Trainer v2), SDG Hub, Eval Hub, KServe

4 Design Principles

MLflow serves as the single source of truth, preventing data drift between the outer and inner loops. Both loops read from the same trace store but produce different artifacts — SKILL.md files vs model checkpoints. This separation allows them to operate independently while sharing a unified data plane.

Eval Hub acts as the single orchestration point for all evaluations — agent skills, model benchmarks, safety, and performance. This avoids evaluation fragmentation where different components use incompatible scoring systems, making it impossible to compare results across the pipeline.

Start with a frontier LLM (e.g., Claude Opus), distill production traces into a tuned open-source model (e.g., Llama 70B), then further compress to a Small Language Model (e.g., Granite 8B). Each stage maintains task performance while reducing inference costs 10–30x. Ambitious but validated by NVIDIA research and OpenAI's distillation results.

The critical differentiator of this architecture. Real production traces reflect actual usage patterns — the prompts users actually send, the tools they actually use, the edge cases they actually hit. Synthetic data augments but never replaces real production data.

All components run natively on OpenShift AI, reducing operational complexity. This means consistent RBAC, monitoring, networking, and resource management. No external SaaS dependencies required for the core loop.
