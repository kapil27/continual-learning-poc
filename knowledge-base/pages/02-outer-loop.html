Outer Loop

Skill Optimization in Token-Space

Optimizes what the agent is told — instructions, prompts, and tool configurations — without changing model weights. The output is always an improved SKILL.md.

1 Pipeline Stages

eval-analyze

Examines a target skill's SKILL.md, discovers test cases, and generates eval.yaml with natural language schema descriptions, suggested judges (inline checks + LLM quality assessment), and regression thresholds.

Output: eval.yaml contains
  • → Natural language schema descriptions of dataset and outputs
  • → Suggested judges (inline checks + LLM quality)
  • → Regression thresholds

eval-dataset

Generates test cases based on skill analysis. Creates input.yaml files with prompts, optional reference outputs, and annotations. Can bootstrap a starter dataset or expand an existing one to improve coverage.

Output: input.yaml per test case
  • → Test prompts matching real usage patterns
  • → Optional reference outputs for comparison
  • → Annotations for ground truth validation

eval-run

The core execution engine. Orchestrated by skills/eval-run/scripts/ — the most complex step in the pipeline.

Sub-step Script Action
Config discoverydiscover.pyFind and load eval.yaml
Preflightpreflight.pyCheck for stale artifacts
Workspace setupworkspace.pyIsolated workspace per case + hook setup
Executionexecute.pyInvoke runner (EvalRunner.run_skill())
Collectioncollect.pyParse stdout → events.json, gather artifacts
Scoringscore.pyRun judges, compute pass rates & means
Reportingreport.pyGenerate analysis and HTML report

eval-review

Human-in-the-loop review. Presents judge scores and skill outputs for qualitative feedback. Produces review.yaml with per-case assessments. This is where human judgment augments automated scoring.

Review workflow
  • 1. Present judge scores alongside raw outputs
  • 2. Reviewer provides qualitative feedback per case
  • 3. Generates review.yaml with assessments
  • 4. Feedback drives eval-optimize decisions

eval-optimize

The closed loop step. Reads judge failures and rationales, performs root cause analysis, then makes surgical edits to SKILL.md only. Re-runs eval-run with --baseline to check for regressions.

Hard Constraints
  • ✗ Does NOT modify test cases, judges, or eval.yaml
  • ✗ Does NOT edit builtin judge code
  • ✓ Only edits SKILL.md (skill instructions/prompt)
  • ↻ Iterates up to max-iterations (default 3)

2 Runner Abstraction

The harness is agent-agnostic via the EvalRunner ABC. Three built-in runners:

Runner runner.type Mechanism Events
ClaudeCodeRunner claude-code claude --print --model {model} --output-format stream-json Full
CliRunner cli Arbitrary command template with {placeholder} substitution If Claude-compat JSONL
ResponsesAPIRunner responses-api OpenAI Responses API + Shell tool Partial

CLI Runner — OpenCode Configuration

The cli runner is the path for running eval-harness with OpenCode or any non-Claude agent. Available placeholders:

Placeholder Source
{agent}System prompt / skill content
{workspace}Isolated workspace directory
{model}Model identifier from eval.yaml
{output_dir}Artifacts output directory
{timeout}Max execution time
{max_budget_usd}Cost budget cap
{effort}Effort level hint
{system_prompt}Custom system prompt override
{args}Extra arguments + input.yaml fields
Example eval.yaml for OpenCode 
runner:
  type: cli
  command: "opencode run {agent} --workspace {workspace} --model {model}"

models:
  - qwen3-30b-a3b

dataset:
  path: ./test-cases/

judges:
  - name: task_completion
    prompt: "Did the agent complete the assigned task?"
    scale: [1, 5]
  - name: tool_usage
    builtin: "cost_budget"

thresholds:
  min_pass_rate: 0.8
  min_mean: 3.5

4 The outputs Dict

Every judge receives this rich record per test case. Click to expand each key:

Dictionary of {path: content} — all files the agent created or modified during the test case execution. Useful for checking if expected output files were generated.

Full event stream from events.json. For Claude Code runner: complete stream-json JSONL with tool calls, assistant messages, system events. For CLI runner: only if stdout produces Claude-compatible JSONL.

Extracted root-level assistant text from the event stream. The "final answer" the agent gave, without tool call details. Primary input for LLM judges evaluating response quality.

Ordered list of tool invocations from the event stream. Includes tool name, arguments, and results. Useful for judges that evaluate agent behavior (e.g., "did the agent use the right tools?").

Quantitative metrics: process exit code (0 = success), wall-clock duration in seconds, total API cost in USD, and conversation turn count. Used by builtin judges like cost_budget.

Ground truth data from annotations.yaml in the dataset. Provides expected outputs, reference answers, or validation criteria that judges can compare against.

5 Judges as DPO Signal

Judge scores map naturally to training signals — this is the bridge between the outer loop and inner loop.

Pairwise Comparison
🏆 → DPO Pairs

compare_runs() with position-debiased LLM judge.
Direct mapping: winner = chosen, loser = rejected.

Bool Judges
✓✗ → Pass/Fail Labels

Pass/fail + rationale per case.
GRPO reward: pass = 1.0, fail = 0.0. Binary filtering for training data.

Numeric LLM Judges
📊 → GRPO Rewards

1–5 scale scores per case.
Continuous reward signal for reinforcement learning. Score ≥4 → chosen, ≤2 → rejected.

6 Eval-Gating Thresholds

Threshold Applies to Failure Condition
min_pass_rateBool judgesPass rate below threshold
min_meanNumeric judgesMean score below threshold
min_win_ratePairwise judgesWin rate below threshold
Baseline comparisonAll judgesMean/pass_rate drop > 0.5 vs baseline
Thresholds Config Example 
thresholds:
  min_pass_rate: 0.8        # Bool judges must pass 80%+
  min_mean: 3.5             # Numeric judges must average 3.5+
  min_win_rate: 0.55        # Pairwise judges must win 55%+
  regression_tolerance: 0.5 # Max allowed drop vs baseline
