Issues Found & Solved

9 issues encountered during deployment — all resolved. Click to expand details.

#1 MLflow SQLite fails under SCC Critical
Symptom 
OperationalError: unable to open database file
mlflow.store.db.utils: /mlflow/mlruns/mlflow.db — read-only file system
Root Cause

OpenShift Security Context Constraints (SCC) prevent arbitrary UID from writing to container filesystem paths. MLflow defaults to SQLite which requires writable local storage.

Fix 
Deploy PostgreSQL 16 as backend store:
MLFLOW_BACKEND_STORE_URI=postgresql://mlflow:***@postgresql:5432/mlflow
#2 psycopg2 demands SSL Medium
Symptom 
psycopg2.OperationalError: connection to server refused
FATAL: no pg_hba.conf entry for host ... SSL off
Root Cause

Default psycopg2 connection attempts SSL negotiation. In-cluster PostgreSQL has no TLS cert configured, causing connection rejection.

Fix 
Append ?sslmode=disable to connection URI:
postgresql://mlflow:pass@postgresql:5432/mlflow?sslmode=disable
#3 HuggingFace xorbs CDN stalls Medium
Symptom 
ConnectionTimeout: cdn-lfs-us-1.hf.co — download stalls at 2.1GB/4.5GB
Pod OOMKilled during inline model download in vLLM init
Root Cause

Downloading multi-GB model weights inside the serving container init causes timeouts and OOM. CDN bandwidth through cluster egress is throttled.

Fix 
Pre-download via Kubernetes Job using huggingface_hub:
huggingface-cli download Qwen/Qwen3-3B --local-dir /models/qwen3-3b
Then mount PVC into vLLM deployment.
#4 RHOAI vLLM image 404 Critical
Symptom 
ErrImagePull: quay.io/modh/vllm:rhoai-2.19 — manifest unknown
Back-off pulling image — ImagePullBackOff
Root Cause

The RHOAI-specific vLLM image tag was not yet published to the registry. Internal image tags have a different lifecycle from the upstream project.

Fix 
Use upstream public vLLM image:
image: vllm/vllm-openai:v0.8.4
#5 Qwen 7B OOMs on T4 Critical
Symptom 
torch.cuda.OutOfMemoryError: CUDA out of memory.
Tried to allocate 512 MiB (GPU 0; 15.89 GiB total; 14.2 GiB reserved)
Root Cause

Qwen 7B requires ~14.5GB VRAM at fp16. T4 has 16GB but after KV cache and overhead, insufficient memory remains for inference at useful context lengths.

Fix 
Downgrade to Qwen3-3B for T4 dry-run:
--model Qwen/Qwen3-3B --max-model-len 32768 --gpu-memory-utilization 0.92
#6 OpenCode run hangs Medium
Symptom 
opencode run "prompt" — process hangs indefinitely
No output, no error, CPU at 0%
Root Cause

OpenCode's TUI mode waits for interactive terminal input even in headless/script mode. Without a proper stdin pipe, it blocks forever waiting for keypress.

Fix 
Pipe stdin to prevent hang:
echo "" | opencode run "your prompt here"
#7 OpenCode URL parse error Medium
Symptom 
Error: invalid URL — cannot parse "http://vllm-svc:8000/v1" as OpenAI endpoint
provider "openai" expects api.openai.com host
Root Cause

OpenCode's default OpenAI provider validates the URL format and rejects non-OpenAI hostnames. A compatible provider type is needed for self-hosted endpoints.

Fix 
Use @ai-sdk/openai-compatible provider in opencode.json:
"provider": { "type": "openai-compatible", "baseURL": "http://vllm-svc:8000/v1" }
#8 vLLM tool calling 400 Medium
Symptom 
HTTP 400 Bad Request: "tool_choice" is not supported
{"error": "This model does not support tool/function calling"}
Root Cause

vLLM does not enable tool calling by default — it requires explicit CLI flags to activate the tool-call parser for compatible models like Qwen.

Fix 
Add vLLM args:
--enable-auto-tool-choice --tool-call-parser hermes
#9 Context overflow 39K > 32K Critical
Symptom 
Error: prompt too long — 39,142 tokens exceeds max 32,768
OpenCode agentic session aborted mid-task
Root Cause

Agentic coding sessions accumulate context rapidly — tool calls, file contents, and conversation history exceed 32K tokens within a few iterations. Qwen 3B on T4 cannot serve longer contexts due to VRAM limits.

Fix 
Production target: Qwen3-32B on A100 (80GB) with 128K context:
--model Qwen/Qwen3-32B --max-model-len 131072 --tensor-parallel-size 2
