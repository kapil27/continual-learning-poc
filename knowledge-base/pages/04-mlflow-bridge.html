Module 04 Data Plane

MLflow as Shared Data Plane

The bridge between production experience and model improvement — how traces become training data.

📊 What MLflow Captures

Data Type Format (Spans) Training Relevance
Prompts & Completions LLM span inputs/outputs ★★★ Direct SFT pairs
Tool Calls TOOL span with args + result ★★★ Function calling fidelity
Token Usage span.attributes token counts ★★☆ Cost analysis only
Latency span.start_time → end_time ★★☆ Perf gating signal
Errors span.status = ERROR ★☆☆ Negative examples (DPO)
Reasoning Chains CHAIN span parent→child tree ★★★ Multi-turn trajectories

🔍 Searching Traces

trace_search.py 
import mlflow

# Connect to MLflow on RHOAI
mlflow.set_tracking_uri("https://mlflow.apps.cluster.example.com")

# Search for successful agent traces from the last 7 days
traces = mlflow.search_traces(
    experiment_names=["agentic-continual-learning"],
    filter_string="status = 'OK' AND timestamp > '2025-05-27'",
    max_results=500,
)

# Each trace contains a tree of spans
for trace in traces:
    # Get the root span (full agent turn)
    root = trace.data.spans[0]
    print(f"Trace {trace.info.request_id}: {len(trace.data.spans)} spans")

    # Extract feedback annotations
    feedback = mlflow.search_assessments(
        trace_id=trace.info.request_id
    )
    for fb in feedback:
        print(f"  → {fb.source.source_type}: {fb.name} = {fb.value}")

⚠️ The Critical Gap: Trace Schema vs Training Data

🌲 MLflow Trace Schema

// OpenTelemetry hierarchical spans
{
"trace_id": "abc123",
"spans": [
{ "span_type": "CHAIN",
"children": [
{ "span_type": "LLM",
"inputs": { messages[] },
"outputs": { choices[] } },
{ "span_type": "TOOL",
"inputs": { tool, args },
"outputs": { result } }
] }
]
]
}

Nested tree of spans with rich metadata. Powerful for observability, but not directly trainable.

📄 Training Data Format

// Flat JSONL — one row per example
{
"messages": [
{ "role": "system",
"content": "You are..." },
{ "role": "user",
"content": "Fix the bug" },
{ "role": "assistant",
"content": "I'll...",
"tool_calls": [...] }
]
}

Flat message arrays needed by transformers training. The tree must be linearized and segmented.

🚨 No existing tool bridges this gap. The trace-to-training converter is the single most critical piece of glue code required.

🔄 Conversion Requirements by Training Objective

Format Requires Gap from MLflow Traces
SFT
Supervised Fine-Tuning
 messages[] array with system / user / assistant roles Medium: Span tree → flat message array. Tool calls must be serialized into assistant turns. Multi-turn boundaries unclear.
DPO
Direct Preference Optimization
 prompt + chosen + rejected completion pairs Large: Need paired traces for same input with different quality. Requires judge scores to rank, then pairing logic. No existing tooling.
GRPO
Group Relative Policy Optimization
 prompt + completions[] group (N samples) + reward per completion Large: Need N completions per prompt. Requires either re-generation or grouping production traces by input hash. Reward function must be extractable.

The OAuth sidecar proxy handles Service Account token exchange transparently, so agent pods can use MLflow APIs without credential management.

📦 MLflow Model Registry

✅ Capabilities

  • ✓ Version tracking with aliases (champion / challenger)
  • ✓ Model lineage: training run → registered model → deployed endpoint
  • ✓ Artifact storage (model weights, configs, tokenizer)
  • ✓ Stage transitions (None → Staging → Production → Archived)
  • ✓ Integration with KServe for serving

❌ Gap: Compression-Chain Lineage

  • ✗ No native concept of "model A was distilled from model B"
  • ✗ Cannot track progressive compression chains (Opus → Llama 70B → Granite 8B)
  • ✗ No comparison dashboards for compression stages
  • ✗ Workaround: use model tags + custom metadata to encode parent model references
